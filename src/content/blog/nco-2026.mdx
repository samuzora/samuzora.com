This year, Singapore's National Cybersecurity Olympiad (NCO) was organized by a
group of CTF players from slight_smile. They had an interesting way of
preventing LLM abuse during the olympiad. For the qualifiers, internet access
was completely blocked and certain sites and documentation was whitelisted and
brought into the intranet for participants to reference. For the finals,
internet access was allowed through a proxy, but LLM use was strictly
disallowed, other than a proxied and lobotomized version of gpt-4 that had a
weakened system prompt and limited token budget per participant. As such, the
challenges had to be toned down accordingly, especially for the qualifiers, to
ensure that the competitors could still reasonably solve a good number of
challenges within the tight time frame of about 5 hours. I set a medium pwn
challenge for qualifiers and 1 medium and 1 hard pwn challenge for finals.
Although I had to nerf the challenges to the appropriate challenge difficulty, I
think the unnerfed versions are probably more interesting for some participants
so I'll be posting them here together with the writeups for them!

This challenge was meant to be the easier medium challenge, but post-nerf for
both challs we determined that this would be harder than the other one. The
challenge is quite basic: we can create and remove tracks, play a track, add the
currently playing track to a linked list of liked songs, and view details about
tracks.

Let's start analyzing the challenge for any potential vulns we can use!

## Classic UAF

```c
void play() {
  Track *track = tracks[get_idx()];
  current_track = track;
  std::cout << "playing <" << current_track->title << ">" << std::endl;
}
```

This is the function that stores a specified track to current_track. This looks
like a classic UAF if current_track isn't cleared when removing a track, and we
indeed see that happening:

```c
void remove() {
  int idx = get_idx();
  Track *track = tracks[idx];

  Track *cur = fav_head.next;
  for (Track *cur = fav_head.next; (cur != NULL && cur != &fav_head); cur = cur->next) {
    if (cur->title == track->title) {
      unlink_track(cur);
      break;
    }
  }

  delete track;
  tracks.erase(tracks.begin() + idx);
}
```

In `remove(){:c}`, the track is freed and removed from the vector of tracks, but
current_track is not cleared. This is our first instance of UAF, and immediately
we can already get our heap and libc leak via status:

```
current tracks:
currently playing <DvbY\x05\x00\x00\x00%\xfc=\xa6\x17\x02\xb58>
(c)reate track
(r)emove track
(l)ike current track
(p)lay track
(s)how status
>
```

This UAF also allows us to add a freed track to the linked list of liked songs.
Let's take a look at how the linked list is implemented.

## Yet Another Linked List

Why are we even using a linked list to store the liked tracks? This seems
neither efficient nor suited for the purpose at all. As it turns out, the linked
list implementation is full of bugs.

### Repeated nodes

```c
void like() {
  if (current_track) {
    current_track->prev = fav_head.prev;
    current_track->next = &fav_head;
    fav_head.prev->next = current_track;
    fav_head.prev = current_track;
    std::cout << "liked <" << current_track->title << ">" << std::endl;
  } else {
    std::cout << "play a song before liking it!" << std::endl;
  }
}
```

As one can see, there is no check on whether the node has already been added to
the linked list. If we like the same track twice and then delete the track, the
track will only be removed from the linked list once, leaving the other node
pointing to a freed track. This is our second instance of UAF. With this UAF, we
can attack the unlink mechanism by re-allocating and removing another track with
the same title.
