As the parasite spread, the system began to evolve - reshaping itself for speed,
efficiency... and control.

One anomaly stands out: Array.prototype.swap, a new native function embedded
deep in the JavaScript engine.

Design to optimize performance, it grants direct access to internal array memory
- dangerously low-level for a high-level language.

But this isn't just an upgrade. It's a vector. A tool hiding in plain sight.
Controlled just right, it bends memory boundaries. Controlled wrong, it bends
reality.

Every keystroke counts. Get inside, manipulate the memory, and trace the glitch
back to its origin.

This is your entry point - and the race to the core begins with a single swap.
</Challenge>

This was a Spidermonkey engine exploitation challenge. The primitive given to us
is quite simple. A new swap method is defined for Array objects. The method
takes in no arguments. The elements to swap are indicated by the `from{:c}` and
`to{:c}` properties defined on the Array object. Below is the implementation of
the swap method.

```cpp
bool js::array_swap(JSContext* cx, unsigned argc, Value* vp) {
  AutoJSMethodProfilerEntry pseudoFrame(cx, "Array.prototype", "swap");
  CallArgs args = CallArgsFromVp(argc, vp);

  // {obj} refers to this == b
  RootedObject obj(cx, ToObject(cx, args.thisv()));
  if (!obj) {
    return false;
  }

  NativeObject* nobj = &obj->as<NativeObject>();
  
  uint64_t len;
  if (!GetLengthPropertyInlined(cx, obj, &len)) {
    args.rval().setBoolean(false);
    return false;
  }

  if (!CanOptimizeForDenseStorage<ArrayAccess::Read>(obj, len)) {
    JS_ReportErrorASCII(cx, "Cannot optimize for array object");
    args.rval().setBoolean(false);
    return false;
  }

  uint64_t capacity;
  capacity = nobj->getDenseCapacity();

  const js::Value* elements2 = nobj->getDenseElements();

  #if defined(DEBUG) || defined(JS_JITSPEW)
    printf("Current object: %p\n", static_cast<void*>(nobj));
    printf("Current length/capacity: %lu / %lu\n",  len, capacity);
    printf("elements: %p\n", elements2);
  #endif


  RootedValue from(cx);
  JSAtom* fromAtom = Atomize(cx, "from", strlen("from"));
  if (!fromAtom) {
    return false;
  }
  RootedId fromId(cx, AtomToId(fromAtom));
  if (!GetProperty(cx, obj, obj, fromId, &from)) {
      args.rval().setBoolean(false);
      return true;
  }

  if (!from.isInt32()){
    JS_ReportErrorASCII(cx, "from is not Int32");
    args.rval().setBoolean(false);
    return true;
  }


  RootedValue to(cx);
  JSAtom* toAtom = Atomize(cx, "to", strlen("to"));
  if (!toAtom) {
    return false;
  }
  RootedId toId(cx, AtomToId(toAtom));
  if (!GetProperty(cx, obj, obj, toId, &to)) {
      args.rval().setBoolean(false);
      return true;
  }

  if (!to.isInt32()){
    JS_ReportErrorASCII(cx, "to property is not Int32");
    args.rval().setBoolean(false);
    return true;
  }
  
  // truncate - should be fine
  uint64_t fromVal = from.toInt32();
  uint64_t toVal = to.toInt32();

  #if defined(DEBUG) || defined(JS_JITSPEW)
    printf("fromVal: %ld\n", fromVal);  
    printf("toVal: %ld\n", toVal);
  #endif


  if (fromVal < len 
      && toVal < len
      && fromVal < capacity    
      && toVal < capacity)    
  {
    Value tmp = elements2[toVal]; 
    Value tmp2 = elements2[fromVal]; 

    #if defined(DEBUG) || defined(JS_JITSPEW)
      printf("To:\n");
      js::DumpValue(tmp);

      printf("From:\n");
      js::DumpValue(tmp2);
    #endif

    memcpy((void*)&elements2[fromVal], &tmp, sizeof(js::Value));
    memcpy((void*)&elements2[toVal], &tmp2, sizeof(js::Value));


    args.rval().setBoolean(true);  
  }
  else
  {
    JS_ReportErrorASCII(cx, "Index larger than length!");
    args.rval().setBoolean(false);  
  }
  
  return true;
}
```

Before we analyze the method, let's take a look at how Arrays are represented in
Spidermonkey. [This article](https://phrack.org/issues/69/14) was very useful in
helping me understand the inner workings of Spidermonkey.

Importantly, we see the following:

- +0x10 (`0x18dd3cf3b160`): pointer to the Array's backing buffer
- +0x1c (`0x00000003`): initializedLength (will discuss this further)
- +0x20 (`0x00000006`): capacity, the maximum capacity of the backing buffer

Capacity and length should be pretty obvious as to what they represent.
Interestingly, length is writable on Arrays as per the JavaScript spec. This
means that we can't fully trust the length property of the Array, as it can be
easily modified to access the array out-of-bounds. Therefore, the authors of
Spidermonkey decided to introduce the initializedLength field, indicating the
true length of an Array.

See the effect of initializedLength:

```js

a.length = 1
a.length = 3
a // outputs [1, , ]
```

The internal memory representation doesn't change, as we can still see the
values 2 and 3 in memory. However, the engine refers to the initializedLength
value to infer that the backing buffer only holds 1 valid element. Therefore,
the 2nd and 3rd elements are considered uninitialized, and their values are
ignored.

In this block, we see that fromVal and toVal are checked against length and
capacity, instead of initializedLength. This means that we can modify length to
access uninitialized values that are within capacity in the Array. This could
prove useful for some sort of use-after-free. To trigger the UAF, we need to
first understand how the Spidermonkey garbage collector works.

## Spidermonkey's garbage collector

In Spidermonkey, there are 2 heap regions, which correspond to 2 cycles of
garbage collection (GC). The first is the nursery which stores short-lived
objects. New objects are first allocated in this nursery heap. When minor GC
occurs, all objects no longer in use will be freed, while any surviving objects
in the nursery will be moved to the tenured heap.

The nursery heap's implementation is quite simple: it's akin to the top chunk
in ptmalloc. Whenever a new object is allocated, a new chunk is always carved
from the remainder chunk as long as there is sufficient space. When there is
insufficient space for a new object, the minor gc cycle is triggered. During the
minor gc cycle, the garbage collector will traverse all objects starting from
the root object. Any object that is reachable and resides in the nursery will be
moved to the tenured heap. If the object holds references to other nursery
objects, these objects will be moved to the tenured heap as well. At the end of
the cycle, there should be no more live objects in the nursery heap. Therefore,
the heap will be reset and can be used for other short-lived objects.

The tenured heap in Spidermonkey is implemented using the jemalloc allocator.
For the purposes of this writeup, it's not as important how the tenured heap
performs its garbage collection.
